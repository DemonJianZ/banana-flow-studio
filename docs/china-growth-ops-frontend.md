# China Growth Ops 前端交互设计

## 目标

把当前的“输入一句话 -> 直接生成 3 个 Topics”的脚本流程，调整成符合 `china-growth-ops-skill` 的半自动运营工作台。

这套交互要解决两个问题：

1. 在生成前锁定商业单元，而不是只拿到一个产品名就开跑。
2. 在生成后保留人工选择和确认，不把高风险动作做成一键直达。

## 当前前端现状

- 当前脚本请求只发送 `product`：
  - `src/api/agentCanvas.ts`
- 当前脚本缺信息时，只追问“你想做哪个产品/品类？”：
  - `src/pages/Workbench.jsx`
- 当前结果卡只展示 `Topics`：
  - `src/components/agent-canvas/TopicCards.tsx`

这和 skill 的执行模型不一致。skill 明确要求：

- 先锁定：产品、目标买家、价格带、转化目标、主平台
- 生成 3 个角度后，由用户选 1 个 winner
- 再生成平台适配内容
- 遇到发布、支付、私信批量触达等不可逆动作时，必须人工确认

## 设计原则

1. 仍然保持聊天式入口，不改成传统后台大表单。
2. 通过“渐进式补全”收集信息，每次只问用户一个小决策。
3. 把结构化选择放进卡片，不让用户反复手打。
4. 默认是“草稿模式”，任何高风险动作都需要单独确认。
5. 结果区不只展示 Topics，要分阶段展示“选择 -> 适配 -> 执行准备 -> 复盘”。

## 推荐交互流程

### 1. 入口阶段：聊天触发

用户输入示例：

- 帮我做一个洗面奶的抖音脚本
- 我要做一个小红书防晒种草方案
- 给我出一个便携理线器的带货脚本

系统先识别：

- 产品
- 平台
- 目标动作是否明确

如果信息不足，不直接失败，而是弹出“脚本设定卡”。

## 2. 脚本设定卡

### 交互形式

在聊天流中插入一张可编辑卡片，不打开新页面，不打断当前会话。

### 字段

- 产品 / 服务
- 目标人群
- 价格带
- 转化目标
- 主平台
- 次平台

### 输入方式

- 产品：文本输入 + 常用品类 chip
- 目标人群：文本输入 + 常用画像 chip
- 价格带：单选 chip
- 转化目标：单选 chip
- 主平台：单选
- 次平台：可选单选

### 默认策略

如果用户没填全：

- 价格带默认“未指定”
- 次平台默认“不选择”
- 目标人群默认“单一核心人群”
- 转化目标默认“点击/咨询”

### 卡片底部动作

- `开始生成`
- `先用默认值`
- `取消`

## 3. 角度候选阶段

当前 `TopicCards` 只负责展示，不支持“选一个 winner”。这里需要改成“角度候选卡”。

### 每个候选卡展示

- 角度类型：人物角度 / 场景角度 / 误区角度
- 标题
- 开头 Hook
- 60 秒正文摘要
- 为什么适合当前平台

### 每个候选卡动作

- `选这个继续`
- `重生成这个角度`

### 区域顶部动作

- `全部重生成`
- `直接用默认推荐`

### 默认行为

如果用户不选，系统可以高亮一个“推荐角度”，但不要自动进入下一步。

## 4. 平台适配阶段

skill 明确要求从一个 source offer 适配到不同平台，所以中间不能只显示 Topics。

### 交互结构

选中 winner 后，结果卡切换成分段结构：

1. 选定卖点
2. 平台计划
3. 内容包
4. 草稿字段
5. 风险与阻塞
6. 复盘指标

### 平台计划

显示：

- 主平台
- 次平台
- 主平台目标
- 内容类型
- CTA 方向

UI 形式：

- 顶部主次平台切换条
- 主平台内容默认展开
- 次平台内容折叠

## 5. 内容包阶段

按 skill 的模板，前端不要只留脚本文本，要拆成可消费模块。

### 内容包模块

- 短视频 Hook
- 口播脚本
- 标题
- 卖点 bullet
- FAQ / 异议处理
- 私聊回复模板

### 交互动作

- `复制`
- `替换重写`
- `按主平台重写`
- `按次平台重写`

对于抖音、小红书、电商平台，展示模块应不同：

- 抖音：Hook、15 秒脚本、CTA 变体
- 小红书：标题、封面字、正文结构、评论引导
- 电商：商品标题、短描述、长描述、FAQ

## 6. 浏览器执行准备

skill 不是要全自动发布，而是“准备好字段并停在人工确认”。

所以前端要有一个“执行准备”区，而不是直接做发布按钮。

### 区域内容

- 页面类型
- 需要填写的字段
- 推荐标签
- 商品标题
- 短描述
- CTA 文案
- 需要人工处理的步骤

### 风险拦截

下面这些动作必须单独确认：

- 发布
- 支付
- 批量私信
- 账号敏感设置

### UI 形式

- 默认只展示“准备草稿”
- 风险动作按钮置灰
- 点击高风险动作时弹确认层：
  - 当前风险
  - 不可逆说明
  - 需要人工确认

## 7. 复盘与下一步

skill 要求每次执行后记录输出和 next action，所以结果卡底部要有“下一步建议”。

### 固定展示

- 本次平台
- 选定卖点
- 已生成内容
- 当前阻塞点
- 下一步人工动作
- KPI 检查项

### KPI 项

- 浏览量
- 点击率
- 收藏 / 保存
- 咨询数
- 加购
- 下单
- 退款信号

### 动作

- `开始下一轮改写`
- `改成另一个平台`
- `沿用同一卖点继续`

## 8. 和当前项目结构的映射

### 前端状态层

建议在 `Workbench.jsx` 新增一个 `scriptBriefDraft` 状态，替代当前只保存 `pendingTask.product` 的做法。

建议字段：

- `product`
- `audience`
- `priceBand`
- `conversionGoal`
- `primaryPlatform`
- `secondaryPlatform`
- `selectedAngle`

### 新增组件

- `ScriptBriefCard`
- `AnglePickerCard`
- `PlatformPlanView`
- `CopyPackView`
- `BrowserReadyFieldsView`
- `RiskBlockersView`
- `KpiChecklistView`

### 替换组件

- 当前 `TopicCards` 不应继续承担最终结果展示职责
- 它应改成“候选角度选择器”，不是只读文本列表

## 9. 对后端接口的最低要求

当前接口只发 `product`，不够支撑这套交互。

`/api/agent/idea_script` 最低应改成接收：

- `product`
- `audience`
- `price_band`
- `conversion_goal`
- `primary_platform`
- `secondary_platform`
- `selected_angle`

返回建议至少包含：

- `angle_candidates`
- `selected_offer`
- `platform_plan`
- `copy_pack`
- `browser_ready_fields`
- `risks_and_blockers`
- `kpi_checklist`

## 10. 最小可落地版本

如果先不动太多后端，建议前端先做第一阶段最小改造：

1. 生成脚本前，补一个“脚本设定卡”
2. Topics 改成“候选角度选择卡”
3. 结果卡新增“平台计划”和“风险提示”两个区
4. 保留现有聊天流和结果卡容器，不改整体布局

这样改动最小，但已经能让交互方式更接近这份 skill 的真实用法。
